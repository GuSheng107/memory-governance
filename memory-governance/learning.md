# Memory Governance Learning Rules

## Learn From

Learn only from:

- explicit user corrections
- explicit user preferences
- repeated successful patterns
- clear lessons after failure
- workflow improvements that help more than one task

## Do Not Learn From

Do not learn from:

- silence
- one-off instructions
- unverified guesses
- noisy environment output
- temporary workarounds
- secrets, tokens, cookies, or personal data

## Promote To Stable Memory When

Promote a lesson from `corrections.md` to `memory.md` only when:

- the same rule has been validated more than once
- the user framed it as a durable requirement
- it repeatedly improves outcomes
- a failure lesson is now clear enough to be a reusable rule

## Keep In Corrections When

Keep the lesson in `corrections.md` when:

- it happened only once
- it still depends on context
- it is not clearly stable yet
- it reads more like a case than a rule

## Compression Hand-off

When memory is bloated, repetitive, or conflict-heavy, hand off the cleanup to `memory-evolution` instead of expanding these files manually.
